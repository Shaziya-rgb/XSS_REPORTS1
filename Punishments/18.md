## Title
Reflected Cross-Site Scripting (XSS) via GET Parameters Reflected Directly into HTML Response

## Vulnerability Type
Reflected XSS

## Summary
The `fname` parameter on the punishment lab page reflects user input directly into the HTML page response without any encoding, escaping, or sanitization. When a value is submitted via the GET parameter, it gets embedded raw into the page's HTML output. This means if you inject HTML/JavaScript tags into the parameter value, they get rendered and executed by the browser immediately. No storage involved — the payload fires the moment the page loads with the crafted URL.

## Vulnerable Endpoint
```
https://kzlabs.com/punishment/1.php?fname=tic&lname=toe
```
**Vulnerable Parameter:** `fname` parameter reflected inside the HTML response body without any output encoding

## Steps to Reproduce
1. Go to `https://kzlabs.com/punishment/1.php?fname=tic&lname=toe` and observe the First Name and Last Name input form.
2. Submit this as the fname value or visit the following URL directly:
```
https://kzlabs.com/punishment/1.php?fname=tic"><script>alert(1)</script><%2F&lname=toe
```
3. The page loads and a JavaScript alert box pops up displaying `1`, confirming the payload broke out of the HTML context and executed.

## Payload Used
```
tic"><script>alert(1)</script>
```

## Impact
- An attacker can craft a malicious URL and send it to any user of the site — the moment they click it, the script executes in their browser
- Steal session cookies of anyone who clicks the link, giving the attacker full access to their account
- Since the vulnerable endpoint visibly shows backend source code, it also leaks implementation details making further exploitation easier
- Can be used to redirect victims to fake login pages to harvest credentials
- The vulnerability exists in the most basic form of PHP output — direct `echo` of GET parameters — which shows no output encoding practices are in place, making it trivially exploitable and easy to miss during code review

---

## Remediation
- Escape all user input before reflecting it into HTML by using `htmlspecialchars()` in PHP with the `ENT_QUOTES` flag so characters like `"`, `<`, and `>` cannot break out of the HTML context.
- Filter out HTML tags like `<script>`, `<img>`, `<svg>` from GET parameters before echoing anything back to the page.
- Filter out JavaScript methods like `alert()`, `confirm()`, `prompt()` so even if a tag slips through the method won't execute.
- Use Cloudflare as they have WAF rules that block almost all common XSS payloads automatically before they even reach the server.
