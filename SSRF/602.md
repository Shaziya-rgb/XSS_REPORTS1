SSRF-602
Title
Server-Side Request Forgery (SSRF) via LinkSnap URL Parameter

Vulnerability Type
Server-Side Request Forgery (SSRF)

Summary
The application has a LinkSnap feature where you paste a URL and the server goes and fetches it to generate a preview. There is no restriction on what URL you can put in there. So instead of a normal website URL i put in a special value that told the server to read its own internal files and send the contents back to me. The server did exactly that without any complaint.

Vulnerable Endpoint
POST /602.php
linksnap_url

Steps to Reproduce

1.Navigate to the following URL
```
http://kzlabs.in/602.php
```
2.In the Link URL field enter the following instead of a normal URL
```
file:///etc/passwd
```
3.Click Generate Preview

4.The server reads its own internal user account file and sends the full contents back in the response

Crafted Request
linksnap_url=file:///etc/passwd

Proof of Concept
Screenshot 1 — The response showing the full contents of the server's own user account file coming back.

<img width="1784" height="959" alt="image" src="https://github.com/user-attachments/assets/2ef5d431-6d88-42fe-a233-112d1f571470" />


Impact
1.Attacker can read sensitive internal files from the server such as user account details, configuration files and potentially private keys
2.If configuration files containing database passwords or API keys are read, full system compromise is possible
3.This can be chained with other attacks to go from file reading to full server takeover

Remediation

1.Add a CSRF token to every form so the server can verify the request came from the real user
2.Check Origin and Referer headers and block anything not coming from your own domain
3.Never use GET requests for actions that change data, always use POST
