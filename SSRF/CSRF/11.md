CSRF-11

Title
CSRF Account Wipe  via Missing Anti-CSRF Token

Vulnerability Type
Cross-Site Request Forgery (CSRF)

Summary
I was testing the Disable 2FA  and noticed the request goes through with no token or session validation attached to it. I went ahead and crafted a malicious HTML page, sent it to the victim and the moment they opened it their. No clicks needed, no forms to fill, just open the page and its done.
```
http://kzlabs.in/13011.php?action=settings

```
Vulnerable Endpoint
POST /13011.php?action=settings

Steps to Reproduce
1.Log into the application at
````

http://kzlabs.in/13011.php?action=settings
````
2.Head over to account settings and Disable 2FA
3.Turn on Intercept in Burp before hitting submit
4.Click Disable 2FA and catch the request in Burp
5.Right click the request and go to Engagement Tools then Generate CSRF PoC
6.Save the HTML file it gives you
7.Open that file in a browser while still logged in
8.Account and data deleted automatically, no warning, no validation, nothing stops it

Crafted Request

```
<html>
  <!-- CSRF PoC - generated by Burp Suite Professional -->
  <body>
    <form action="http://kzlabs.in/1311.php?action=disable2fa" method="POST">
      <input type="submit" value="Submit request" />
    </form>
    <script>
      history.pushState('', '', '/');
      document.forms[0].submit();
    </script>
  </body>
</html>
```

Impact

1.Complete account takeover
2.Victim gets no notification, no popup, no warning at all

Remediation

1.Generate a unique CSRF token per session and tie it to every password change request
2.Reject any request where the token is missing or does not match
3.Cross check the Origin and Referer headers and drop anything that did not originate from your own domain
